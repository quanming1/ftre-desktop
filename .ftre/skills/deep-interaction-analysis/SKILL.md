# Deep Interaction Analysis

通过模拟用户场景，系统化分析交互设计的合理性。

## 分析步骤

### 1. 识别核心概念

**方向**：找出交互中容易混淆的状态和概念

- 列出所有"选中"类状态（active/focused/selected/hover）
- 明确每个状态的触发方式和视觉表现
- 检查状态之间是否冲突或覆盖
- 用户能否直观区分这些状态

### 2. 场景模拟

**方向**：用固定模板描述用户操作流程

```markdown
### 场景 N：[场景名称]

**用户操作：**
1. 用户做了什么
2. 然后做了什么

**当前行为：**
[代码逻辑 → 实际结果]

**用户反应：** 😊/🤔/😤
> "用户可能说的话"

**问题/建议：**
- 发现的问题或改进点
```

### 3. 场景清单

**方向**：确保覆盖主要交互路径

- **导航类**：点击选中、键盘导航、搜索定位、自动跟随
- **操作类**：新建、重命名、删除、拖拽移动
- **状态类**：切换上下文、刷新、展开/收起
- **边界类**：空状态、大量数据、快速连续操作

### 4. 状态同步检查

**方向**：检查多个状态源是否一致

- UI 状态与数据状态是否同步
- 父子组件的状态是否一致
- 操作完成后相关状态是否正确清理
- 上下文切换时旧状态是否残留

### 5. 焦点与键盘

**方向**：键盘操作是否符合预期

- DOM 焦点与逻辑焦点是否对应
- 快捷键在不同焦点区域是否生效
- Tab 顺序是否合理
- 焦点丢失后能否恢复

### 6. 竞品对比

**方向**：与成熟产品对比找差距

- 列出对标产品（VS Code、WebStorm 等）
- 逐项对比功能支持情况
- 标注可配置项 vs 固定行为
- 识别用户习惯预期

## 问题分级

| 级别 | 符号 | 描述 |
|------|------|------|
| 严重 | 🔴 | 功能失效、数据丢失、无法完成任务 |
| 中等 | 🟡 | 体验不佳但可用、需要额外步骤 |
| 轻微 | 🟢 | 可优化的细节、锦上添花 |

## 改进分类

| 周期 | 描述 |
|------|------|
| 短期 | 可立即修复的小问题，几行代码 |
| 中期 | 需要一定工作量，可能涉及多个文件 |
| 长期 | 需要架构调整或新增大功能 |

## 输出产物

1. **分析文档** — `docs/xxx-interaction-analysis.md`
2. **代码修复** — 针对发现的问题
3. **注释说明** — 记录设计决策的原因

## 症状速查

| 症状 | 排查方向 |
|------|----------|
| 点击后状态丢失 | 状态同步、上下文切换 |
| 界面自动跳转 | 自动跟随逻辑、副作用触发 |
| 快捷键无响应 | DOM 焦点、事件捕获 |
| 操作后位置错乱 | 滚动状态、焦点状态残留 |
| 状态显示不一致 | 多状态源冲突、样式优先级 |
| 重复触发操作 | 事件冒泡、依赖数组 |

## 关键原则

1. **用户视角优先** — 始终从用户角度思考，不是开发者角度
2. **模拟真实场景** — 不是假设，而是真实的操作流程
3. **记录用户反应** — 用 emoji 和引述直观表达用户感受
4. **代码与体验对应** — 每个行为都要找到对应的代码
5. **量化问题严重度** — 不是所有问题都需要立即修复
